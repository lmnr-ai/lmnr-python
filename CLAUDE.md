# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

Laminar Python SDK - tracing and evaluation framework for LLM applications. Provides automatic instrumentation for LLM providers (OpenAI, Anthropic, Groq, etc.), manual tracing via decorators, and an evaluation framework for testing LLM outputs.

## Commands

```bash
# Install dependencies with all extras
uv sync --all-extras --dev

# Run all tests
uv run pytest

# Run specific test file
uv run pytest tests/test_initialize.py

# Run specific test
uv run pytest tests/test_initialize.py::test_laminar_initialize_url_parsing -v

# Lint
uv run flake8 src/

# Format
uv run autopep8 --in-place --aggressive src/lmnr/**/*.py

# CLI commands
lmnr eval <file.py>           # Run evaluations
lmnr datasets pull <id>       # Pull dataset
```

## Architecture

### Core SDK (`src/lmnr/sdk/`)
- `laminar.py` - Main `Laminar` class with `initialize()` entry point and span management
- `decorators.py` - `@observe()` decorator for manual instrumentation
- `evaluations.py` - `evaluate()` function and `Evaluation` class for running tests
- `client/` - HTTP clients (`LaminarClient`, `AsyncLaminarClient`) for API interactions

### OpenTelemetry Layer (`src/lmnr/opentelemetry_lib/`)
- `tracing/instruments.py` - `Instruments` enum listing all auto-instrumentable providers
- `tracing/__init__.py` - `TracerWrapper` singleton managing OTEL setup
- `opentelemetry/instrumentation/` - Provider-specific instrumentors (anthropic/, openai/, groq/, etc.)
- Each provider instrumentor has: `__init__.py` (instrumentor class), `utils.py` (parsing), `span_utils.py` (span creation)

### CLI (`src/lmnr/cli/`)
- `evals.py` - `lmnr eval` command

### Debug mode / replay (`src/lmnr/sdk/debug/`)
- A "debug run" is a normal process started with `LMNR_DEBUG*` env vars; there is no cache server, SSE, orchestrator, or `lmnr dev` command. **v2 has NO in-process cache**: every live LLM call hashes its input messages and asks the **app-server** replay-cache endpoint (`POST /v1/rollouts/{session_id}/cache`) what to do (HIT / MISS / LIVE — shared spec §7). The v1 machinery (`ReplayCache`, `spine.py`, `source_trace.py`, per-path occurrence counters, synchronous cache build, SQL source-trace fetch, span-path matching) is **deleted** — do not look for it.
- This is a **cross-language parity surface** with the TS SDK `lmnr-ts/packages/lmnr/src/debug/`. Keep `config.py`, `hash.py`, `pointer.py`, `replay.py`, `__init__.py` line-comparable. The shared vector `tests/data/debug/input_hash_cases.json` (and `config_truth_table.json`) must stay byte-identical with `lmnr-ts/packages/lmnr/test/data/debug/`; change them in lockstep across both repos.
- Parity invariants that MUST match the TS SDK: truthy set `["true","1","yes","on"]`; `replay_enabled = replay_trace_id is not None and cache_until_span_id is not None`; pointer key order `[trace_id, session_id, replay_trace_id, cache_until, debugger_url, started_at]`; `CONSOLE_PREFIX = "LMNR_DEBUG_RUN "`.
- **Input hash (`hash.py`, shared spec §5):** each cached LLM call is addressed by `hex(blake3(canonical_json(messages_without_system)))`. `_canonical_json` reproduces app-server's `input_dedup.rs::canonical_json` byte-for-byte (object keys sorted lexicographically + recursively, array order preserved, scalars via `json.dumps(..., ensure_ascii=False)`); system-message exclusion mirrors `prompt_hash.rs::extract_system_message` (first `role=="system"`, text pulled from string / content-array / `parts` shapes, dropped only when non-empty). **`blake3` (`>=1.0.0,<2.0.0`) is a new runtime dependency** (added to `pyproject.toml`). Number canonicalization is deferred (`1.0` ≠ `1`); revisit only in the shared spec on both sides. Pinned by `input_hash_cases.json` (`tests/test_debug_replay.py::test_input_hash_matches_shared_vector`).
- **The hash is computed off the live span's `gen_ai.input.messages` attribute** (a JSON STRING every provider stamps before the rollout wrapper runs), read via `input_messages_from_span` in `replay.py`. That attribute is exactly what becomes `spans.input` server-side, so hashing it gives the SAME hash app-server computes from the stored span — parity without re-deriving messages per provider. A missing / unparseable / non-list attribute → `None` → run live (never hash a partial input).
- **Three cache outcomes (`outcome.py` `CacheOutcome(kind, cached)`):** HIT (`{"outcome":"hit","response":{...}}`) → serve the cached span; MISS (`{"outcome":"miss"}`) → latch the process-wide run-live flag so every later call skips the endpoint and runs live; LIVE (`{"outcome":"live"}`, warmup-timeout / transport degrade) → run THIS call live WITHOUT latching. COLD is never seen by the SDK. The HIT `response` is **CachedSpan-shaped** (`attributes`, `output`, `start_time` ns, `end_time` ns) — the provider `cached_response_to_*` reconstructors read `start_time`/`end_time` (ns) plus `attributes["gen_ai.output.messages"]` / `attributes["lmnr.sdk.raw.response"]` and reset usage to 0 tokens.
- **The cache HTTP never raises:** `RolloutSessions.cache(session_id, replay_trace_id, cache_until, input_hash)` (sync `rollout_sessions.py` + async mirror) POSTs `{replayTraceId, cacheUntil, inputHash}` and maps the discriminated `{outcome}` body via `_parse_cache_outcome`; any non-2xx or transport exception degrades to `kind="live"`. Only a real MISS latches run-live — that's the **caller's** job (`cache_outcome_for`), not the resource's.
- **A HIT MUST carry a non-null `response` to be servable** — `_parse_cache_outcome` degrades a HIT whose body omits/nulls `response` to `kind="live"` (run this call live, no latch). Without this guard `CacheOutcome(kind="hit", cached=None)` reaches the provider wrappers, which call `cached_response_to_*(None)` → `None.get("type")` → `AttributeError`, and since the chat wrapper re-raises (it is NOT `@dont_throw`) the whole debug run aborts. Fixed once at the shared parse boundary, not in each of the four `cached_response_to_*` methods. Pinned by `test_parse_outcome_hit_without_response_degrades_to_live`.
- **The run-live latch is a process-wide `Laminar.__debug_run_live` flag** (`is_debug_run_live` / `set_debug_run_live`), shared by the sync and async paths. `cache_outcome_for` / `acache_outcome_for` (`replay.py`) short-circuit to `CacheOutcome(kind="live")` WITHOUT hitting the endpoint once it's latched. `Laminar.shutdown()` resets it to `False` so a later `initialize()` starts clean. Tested in `tests/test_debug_replay.py` (`test_cache_outcome_miss_latches_run_live`, `..._live_does_not_latch`, `..._short_circuits_when_already_live`) and `tests/test_debug_runtime.py` (`test_shutdown_resets_run_live_latch`).
- **`replay_enabled()` gates on `get_runtime().replay_configured`, NOT on `get_runtime() is not None`.** A debug-no-replay run (`LMNR_DEBUG` set but no `LMNR_DEBUG_REPLAY_TRACE_ID` / `LMNR_DEBUG_CACHE_UNTIL`) still creates a runtime — needed to stamp `rollout.session_id` and emit the run pointer — but replay must stay off there, or the provider wrappers hit the cache endpoint for a run with no source trace. In v2 `DebugRuntime.replay_configured` collapses to `config.replay_enabled` (there is no synchronous build and no async load window, so no "configured but not ready" state to model). Tested in `tests/test_debug_replay.py` (`test_replay_enabled_false_for_debug_no_replay_runtime`) and `tests/test_debug_runtime.py` (`test_replay_configured_reflects_config`).
- **`LMNR_DEBUG_CACHE_UNTIL` is a span id ONLY — v2 dropped the v1 integer count form.** `config.py` `_parse_cache_until_span_id` → `_normalize_span_id` produces the hyphen-stripped lowercase hex needle `cache_until_span_id` (≤32 chars; `None` when unset or not span-id-shaped, with a warning). All four user forms suffix-match the same span server-side: full UUID `00000000-0000-0000-0123-456789abcdef`, last-two-groups `0123-456789abcdef`, raw 16-hex `0123456789abcdef`, short suffix `abcdef`. The needle is sent **verbatim** as `cacheUntil` in the cache request — app-server does the suffix match against the source trace; the SDK never resolves it (no spine, no occurrence count). There is no `config.cache_until` field anymore. Tested in `tests/test_debug_config.py` + `tests/data/debug/config_truth_table.json`.
- **Two retained clients, closed at shutdown (NOT at init).** `DebugRuntime(config, client, async_client, debugger_url)` holds BOTH a sync `LaminarClient` and an async `AsyncLaminarClient` for the whole run — sync providers look up through `runtime.client`, async providers through `runtime.async_client`. This is the key v1→v2 change: v1 closed the client at the end of init; v2 `Laminar._init_debug_runtime` constructs and **retains** both, and `Laminar.shutdown()` closes them (`runtime.client.close()` + `_close_debug_async_client(runtime.async_client)`, the latter `asyncio.run(async_client.close())`). The bail-out path inside init still closes both. Tested in `tests/test_debug_runtime.py` (`test_runtime_retains_both_clients`, `test_init_registers_session_with_backend` asserts `spy.closed is False` + `runtime.client is spy`, `test_shutdown_closes_retained_clients`).
- **Per-provider async dispatch differs — know which helper each wrapper calls.** The wrapper files are still named `rollout.py` (`instrumentation/{openai,anthropic,google_genai,litellm}/rollout.py`) and import from `lmnr.sdk.debug.replay`. **openai** and **google_genai**: the instrumentation checks `inspect.iscoroutine(result)` and the wrapper returns the coroutine on the async path (`_awrap_*` awaits `acache_outcome_for`), else `cache_outcome_for` sync. **anthropic**: the async call site ALWAYS awaits, so `wrap_create(is_async=True)` returns the `_awrap_create` coroutine directly. **litellm**: sync-only — its instrumentation routes both sync and async calls through the same sync wrapper and only `inspect.iscoroutine`s the result, so the cache lookup is a **blocking** HTTP call even on the async path (a documented v1 limitation carried forward; the lookup is small and one-shot). `get_*_rollout_wrapper()` returns `None` when `replay_enabled()` is false, so the wrappers are inert on non-replay (and non-debug) runs.
- **`mark_span_cached(span)`** stamps the CACHED boundary attributes the frontend renders: `{"lmnr.span.type":"CACHED","lmnr.span.original_type":"LLM"}` — only when the span `is_recording()`, no-op otherwise / on `None`. Called by each provider wrapper right before serving a HIT.
- The pointer's **`started_at` is captured at `DebugRuntime` construction (SDK init), NOT inside `build_pointer`/`emit_pointer`** (the pointer is emitted at shutdown, so reading the clock then would stamp process-exit time). `__init__` records `self._started_at` and passes it to `build_pointer(..., started_at=...)`. `emit_pointer` persists `cache_until=self._config.cache_until_span_id` (the needle as-is — a later `LMNR_DEBUG_FROM_LAST_RUN` re-sends the same needle). TS mirrors via `DebugRuntime._startedAt` → `buildPointer({ startedAt })`. Tested in `tests/test_debug_runtime.py` (`test_emit_pointer_persists_cache_until_span_id`).
- `init_debug_runtime(client, async_client, debugger_url=None)` is one-shot (the `_initialized` global makes it idempotent), but `Laminar.shutdown()` supports a later `initialize()`, so it MUST call `reset_debug_runtime()` to clear `_runtime`/`_initialized` — otherwise a second `initialize()` resurrects the first run's stale session metadata / spent pointer and never re-reads `LMNR_DEBUG*`. When `build_debug_config()` returns `None` (debug off), init does NOT latch `_initialized`, so a later env-flip-on still works. `reset_debug_runtime` is the public reset (tests use it too); TS mirrors `resetDebugRuntime`.
- **The debug `atexit` pointer hook must be unregistered on `shutdown()` or it leaks the retired runtime across init/shutdown cycles.** `_init_debug_runtime` does `atexit.register(runtime.emit_pointer)` so the pointer is emitted at process teardown. `atexit` holds a *strong* reference, so even though `emit_pointer` is one-shot (`_emitted`), an `initialize()`/`shutdown()` loop (test suites, notebooks) keeps every retired `DebugRuntime` — and its retained clients — alive and accumulates dead handlers. Fix: keep the bound method by reference in `Laminar.__debug_exit_hook`, `atexit.unregister` any prior hook before registering a new one, and `atexit.unregister` + null it in `shutdown()`. The synchronous `runtime.emit_pointer()` call at the TOP of `shutdown()` MUST be wrapped in its own `try/except`: `pointer.py` does an UNGUARDED `print(..., flush=True)` (only `_write_pointer_file` is best-effort), and a closed stdout (daemons, broken pipes, notebook restarts) raises `BrokenPipeError`/`OSError` that would otherwise skip `TracerManager.shutdown()` + `reset_debug_runtime()` + the atexit unregister + client close. The guard lives at the `shutdown()` call site (not in `pointer.py`) so `pointer.py`/`pointer.ts` stay line-comparable. `laminar.py` is NOT part of the line-parity surface, but TS has the same fix in `laminar.ts` (`Laminar.debugExitHook` + `process.removeListener`).
- **A debug run registers its SDK-minted session id with the backend** via `client.rollout_sessions.register(session_id)` (`POST /v1/rollouts/{session_id}`, body `{"name": null}`, project-API-key auth). Without it a bare `LMNR_DEBUG=true` run mints a session id the backend never learns about, so the run is invisible in the UI. The register call lives in `Laminar._init_debug_runtime` and is **best-effort** (own inner `try/except` inside the method's outer try/except) — a backend error only warns, never crashes `initialize()`. Python's `initialize()` has no event loop, so this is a direct blocking call (TS does fire-and-forget `void ...catch`). The endpoint is an idempotent upsert keyed on the SDK id; a null `name` never clobbers a name set elsewhere. `register` returns the backend-resolved `projectId` (`response.json().get("projectId")`, `None` on parse failure) so the SDK can build the human-facing URL. The full-URL construction is **one shared code path** `DebugRuntime.debugger_session_url()` — used by BOTH the `cls.__logger.info` console line AND the pointer's `debugger_url` field; after `register`, `_init_debug_runtime` calls `runtime.record_project_id(project_id)` (first-wins) so it returns `<base>/project/<project_id>/debugger-sessions/<session_id>`, else the bare base, else `None`. `debugger_url = os.getenv("LMNR_FRONTEND_URL") or get_frontend_url(base_url)`. For INFO to emit, `_initialize_logger` sets `cls.__logger.setLevel(get_level_from_env())`. The `RolloutSessions` resource carries `register` + `delete` + **`cache`** on BOTH the sync and async clients; debug init uses the sync `register`, and the cache lookups use whichever client matches the provider's sync/async path. TS mirrors `debuggerSessionUrl()` / `recordProjectId`. Tested in `tests/test_debug_runtime.py` (`test_init_registers_session_with_backend` / `test_init_survives_registration_failure` / `test_init_logs_debugger_url_when_project_id_returned`).
- **`LMNR_DEBUG_FROM_LAST_RUN` (truthy) seeds the config from the previous run's pointer file** instead of requiring the ids be copied into env vars. When set, `build_debug_config` calls `_load_last_run()` (reads `${CWD}/.lmnr/last-run.json`, best-effort `{}` on any error) and maps the file's `trace_id` → this run's `replay_trace_id`, reuses its `session_id` and `cache_until` (the stored needle, re-parsed via `_parse_cache_until_span_id`). Individual `LMNR_DEBUG_*` env vars still override per-field. `_load_last_run` imports `POINTER_DIR`/`POINTER_FILE` from `pointer.py` (config→pointer import; pointer.py does NOT import config.py, so no cycle). Keep line-comparable with TS `loadLastRun`. Tested in `tests/test_debug_config.py` (`test_from_last_run_*`).
- Debug tests (`tests/test_debug_*.py`) are isolated unit tests that do NOT need VCR cassettes — they use `monkeypatch` for env vars and `_FakeSpan`/`_FakeRuntime`/`_RecordingCache`/`_SpyDebugClient` doubles. `replay.py` does a function-local `from lmnr.sdk.laminar import Laminar` inside `cache_outcome_for`/`acache_outcome_for` to avoid the `laminar` ↔ `debug` import cycle.
- **The replay cache keys off `gen_ai.input.messages`, which app-server PREFERS over the legacy `gen_ai.prompt.{N}.*` reconstruction.** In `app-server/src/traces/spans.rs::parse_and_enrich_attributes`, the `gen_ai.input.messages` branch (and `gen_ai.output.messages`) OVERWRITES whatever the `gen_ai.prompt.{N}.*` / `ai.prompt.messages` handlers put on `self.input`/`self.output`. So a provider that emits ONLY the legacy per-index attrs has no hashable input for the cache. Both the chat-completions and LiteLLM-responses paths already stamp `gen_ai.input.messages`; the OpenAI **Responses-API** wrapper historically stamped only `gen_ai.prompt.{N}.*`, so it was invisible to replay until we added `build_genai_input_messages` / `build_genai_output_messages` in `instrumentation/openai/v1/responses_wrappers.py` (stamped from `set_data_attributes`, alongside — not replacing — the legacy attrs that the existing `test_responses.py` asserts on).
- **OpenAI Responses-API replay is wired differently from chat/litellm because its instrumentation calls `wrapped()` BEFORE any span exists** (it accumulates a module-level `responses[response_id]` `TracedData` and only mints the span post-hoc when `status=="completed"`). The chat path opens the span in `safe_start_span` and hands it to the rollout wrapper; the responses path can't. So `responses_wrappers.py` keeps its OWN replay orchestration (`_replay_response_sync` / `_replay_response_async`, gated on `_replay_enabled() and not kwargs.get("stream")`) that: opens the span up front, stamps `gen_ai.input.messages` via `build_genai_input_messages(process_input(kwargs["input"]))`, probes `cache_outcome_for`/`acache_outcome_for`, and on HIT calls `OpenAIRolloutWrapper.cached_response_to_responses(outcome.cached)` (in `instrumentation/openai/rollout.py`) to rebuild an `openai.types.responses.Response`, then `set_data_attributes` + `mark_span_cached` + `span.end()`. On MISS/LIVE it runs `wrapped()` live on that same span and records `lmnr.sdk.raw.response`. **Hash parity is guaranteed BY CONSTRUCTION** because the recording run (`set_data_attributes`) and the replay pre-stamp (`_open_replay_span`) build `gen_ai.input.messages` through the identical `build_genai_input_messages(process_input(...))` chain, and app-server stores that string verbatim and re-hashes it. `Responses.retrieve` shares the same wrapper but carries no `input`, so the cache probe returns None and it falls through to live. **Streaming responses are intentionally NOT replayed** (`responses_wrappers.py` still has no streaming span support at all). Only `OpenAIRolloutWrapper.cached_response_to_responses` lives in `rollout.py`; there is no `wrap_responses` there because the responses instrumentation owns its orchestration. Tested in `tests/test_debug_replay_openai_responses.py`.
- **A debug run propagates its coordinates downstream as ONE nested `debug` block of the serialized `LaminarSpanContext`** (`DebugContext` in `sdk/types.py`: `enabled`, `session_id`, `replay_trace_id`, `cache_until`). `LaminarSpanInterfaceMixin.get_laminar_span_context()` stamps it via the module-local `_current_debug_context()` (reads `get_runtime()`, best-effort → `None` when debug is off; only `enabled=True` blocks are ever produced — we are the sole producer). `LaminarSpanContext.deserialize` parses it (accepting camelCase + snake_case; `session_id`/`replay_trace_id` normalized to hyphenated UUID via `_normalize_uuid_like`, `cache_until` kept VERBATIM as the needle; a malformed id drops to `None` rather than raising). Backward-compatible: old SDKs never send it, the block is usually absent in production. This is a **cross-language parity surface** — keep `DebugContext` / `build_debug_config_from_context` line-comparable with TS `deserializeDebugContext` / `buildDebugConfigFromContext`.
- **A downstream service arms — and then REFRESHES — its debug runtime from that propagated block**, so it joins the upstream run regardless of how its spans originate. `Laminar._arm_debug_runtime_from_context(debug)` is called from BOTH `start_span` and `start_as_current_span` right after `_parse_parent_span_context` — i.e. deep in span creation, BEFORE any caching instrumentation consults the runtime, but AFTER the context is parsed. **The coordinates carried in the block (`session_id` / `replay_trace_id` / `cache_until`) are DYNAMIC, not first-wins**: a long-lived downstream service handling many requests must follow each request's context, not freeze on the very first one it ever saw. So the **stable transport (the two retained clients) is separated from the dynamic coordinates** — the clients are built once on first arm and reused; only the coordinates move per request. The arming logic: (a) no runtime yet → build one from the context (latch `_initialized`); (b) a `local_origin=True` (env) runtime exists → env config wins, leave it untouched (the local process owns the run; a propagated context must NOT hijack it) — `_arm_debug_runtime_from_context` bails before any allocation when `existing.local_origin`; (c) a `local_origin=False` (context-armed) runtime exists → REUSE its `client`/`async_client` and `DebugRuntime.update_context_config(config)` refreshes only the dynamic coordinates in place (`local_origin` / `session_minted` are run IDENTITY and never change; `DebugConfig` is frozen so the whole config is rebuilt and the reference swapped under the GIL). `init_debug_runtime_from_context` returns a `(runtime, session_changed)` tuple — `session_changed` is True when the refresh moved to a different session id (or a fresh runtime was just built). The caller (re-)registers the session + re-stamps `rollout.session_id` (onto `__global_metadata` AND `CONTEXT_METADATA_KEY`) ONLY when `session_changed` — a steady stream of requests on the SAME session must not spam the backend or rewrite metadata on every span. `init_debug_runtime_from_context` does NOT latch `_initialized` when the block is absent/unarmed, so a later valid context can still arm. The from-context config is `local_origin=False`: a downstream run reuses the upstream session and may consult the cache, but does NOT open the browser, log the session URL, or register an exit-time pointer hook (the origin owns those). Because the from-context path has no access to `initialize()`'s args, `initialize()` captures `base_url`/`http_port` into `Laminar.__base_url_for_debug` / `__http_port_for_debug` (set unconditionally, even when local debug is off) so it can build its own clients on first arm. The `_init_lock` still serializes the build/refresh so concurrent worker threads don't both build a runtime (loser's clients would leak) or interleave a refresh. **Cross-language parity** — mirrors TS `_armDebugRuntimeFromContext` / `initDebugRuntimeFromContext` / `DebugRuntime.updateContextConfig` (TS holds ONE `rolloutSessions` handle reused via the `rolloutSessions` getter; Python holds the two clients). Tested in `tests/test_debug_runtime.py` (`test_update_context_config_*`, `test_init_from_context_refreshes_context_runtime_reusing_clients`, `test_init_from_context_never_overrides_env_origin_runtime`, `test_arm_from_context_refreshes_session_on_new_context`, `test_arm_from_context_reuses_clients_on_same_session`).
- **A context runtime that races ahead of `_init_debug_runtime` is preempted when `LMNR_DEBUG` is set.** `initialize()` flips `__initialized = True` BEFORE calling `_init_debug_runtime`, and the span funnels gate only on `is_initialized()` — so in that window a span carrying a propagated `debug` block can arm a `local_origin=False` context runtime. `init_debug_runtime` is idempotent, so it would then return THAT runtime; its clients differ from the env-built ones, the `runtime.client is not client` guard fires, and `_init_debug_runtime` bails — leaving the local `LMNR_DEBUG` run with NO session registration, browser open, or pointer hook. Per the PR's "env-origin config owns the process" rule, `_init_debug_runtime` checks `get_runtime()` right after the `LMNR_DEBUG` gate: if a non-`local_origin` runtime exists, it closes that runtime's orphaned clients and calls `reset_debug_runtime()` so the env path below builds a fresh `local_origin=True` runtime. The `runtime.client is not client` guard remains as a defensive fallback for the residual race between the discard and the rebuild. This guard is Python-only on this surface — TS `_initDebugRuntime` instead *rebinds* a pre-armed runtime's rollout-sessions handle (`rebindRolloutSessions`) because a TS from-context runtime can be armed even BEFORE `initialize()` (TS span funnels have no init guard). Maintainer endorsed the guard on the PR ("super unlikely? But we can guard against that, yes"). Tested in `tests/test_debug_runtime.py` (`test_init_preempts_context_runtime_when_env_debug_set`).
- **The one-time browser open is gated on the resolved config, not env vars.** `DebugConfig` gained `local_origin` (origin vs inherited) and `session_minted` (fresh vs reused/provided session id); `DebugRuntime.should_open_browser` is `local_origin and session_minted`. `_init_debug_runtime` now opens the browser on `if runtime.should_open_browser:` (was `if not os.getenv("LMNR_DEBUG_SESSION_ID"):`) — a reused session id (continuation/replay) or a context-armed downstream run no longer reopens the browser. Tested in `tests/test_debug_config.py` (DebugContext parse, from-context builder, local_origin/session_minted, nested-debug `LaminarSpanContext` parse).
- **Reconstructed cached Responses MUST get a UNIQUE id** (`cached_response_to_responses` builds `id=f"cached_{uuid4().hex}"`, NOT a constant `"cached"`). The Responses instrumentation keys its module-level `responses[response_id]` accumulator by id, and `_build_traced_data` prefers the *stashed* `existing_data["input"]` over the live `kwargs["input"]`. With a constant id, the FIRST cached HIT's (shortest) input is stashed under `"cached"`, then every later cached HIT collides on that key and `set_data_attributes` RE-STAMPS `gen_ai.input.messages` with that stale input — overwriting the correct value `_open_replay_span` already set. Symptom: cached Responses spans 2..N all show only span 1's input history (missing tool calls / function_call_output). Note this is the `gen_ai.input.messages` attribute being clobbered, NOT the legacy `gen_ai.prompt.N.*` path — dropping the legacy path does NOT fix it. The chat-completions path is unaffected (it serves the cached response directly and never re-stamps from the accumulator). Pinned by `test_cached_genai_responses_have_unique_ids`.

## Key Patterns

- **Singleton**: `Laminar` and `TracerWrapper` are singletons initialized once at startup
- **Provider instrumentors** extend OpenTelemetry's base instrumentor and are dynamically initialized based on `Instruments` enum
- **Tests use VCR.py** to record/replay HTTP responses in `tests/cassettes/`
- **Tests use InMemorySpanExporter** configured in `tests/conftest.py` for span assertions

## Dev dependencies

- All entries in the `[dependency-groups].dev` section of `pyproject.toml` MUST be pinned to a specific version with `==X.Y.Z`. Do NOT use unbounded specifiers (`>=`, `^`, `~`, `<`, ranges, or bare package names). Pinning keeps the test matrix deterministic across developers and CI. When adding a new dev dep, look up the current release on https://pypi.org and pin to that exact version; bumps then go through a normal PR.

LMNR_PROJECT_API_KEY  # API key (can also pass to initialize())
LMNR_BASE_URL         # API base URL (default: https://api.lmnr.ai)

## Instrumentation tests (VCR)

- Tests under `tests/test_instrumentations/**` replay via VCR cassettes in `cassettes/<module-name>/`. Sensitive headers are filtered by the root `vcr_config` (`authorization`, `api-key`, `x-api-key`, `x-goog-api-key`).
- The sandbox sets `ANTHROPIC_BASE_URL` / `ANTHROPIC_BEDROCK_BASE_URL` to a local proxy; when recording new anthropic cassettes, **unset those two env vars** first or the cassette will record `http://127.0.0.1:...` and fail to replay against `api.anthropic.com`. Use `unset ANTHROPIC_BASE_URL ANTHROPIC_BEDROCK_BASE_URL` before running `pytest --record-mode=once`.
- Re-record a single module with `rm -rf tests/.../cassettes/<module>/ && pytest <path> --record-mode=once`. VCR's default `record_mode` is `once`, so replays work without flags as long as the cassette exists.

## Anthropic instrumentor

- `messages.parse` (the structured-output helper introduced in `anthropic>=0.59`) is wrapped with the same `_wrap` / `_awrap` that wrap `messages.create` — the resulting span name is `anthropic.chat` and the attribute set is identical, plus `gen_ai.request.structured_output_schema`.
- Schema extraction lives in `span_utils._extract_structured_output_schema`. It handles: (1) `output_format=PydanticModel` via `model_json_schema()`, (2) `output_format=<any TypeAdapter-compatible type>` via `pydantic.TypeAdapter(...).json_schema()`, (3) raw-dict `output_format` (bare schema or `JSONOutputFormatParam`), and (4) `output_config={"format": {"type": "json_schema", "schema": {...}}}` on `messages.create`.
- `anthropic.lib.bedrock._beta_messages.Messages` does **not** expose `parse` (only regular + `beta.messages.messages` do). `_instrument` already swallows `ModuleNotFoundError` / `AttributeError` so the missing bedrock.parse attempt is a silent no-op.

## Evaluation trace type propagation

- `_evaluate_datapoint` in `evaluations.py` must set `trace_type=TraceType.EVALUATION` on the isolated context (via `set_association_prop_context`) **before** starting the root evaluation span. Without it, the `LaminarSpanProcessor.on_start()` handler has no `CONTEXT_TRACE_TYPE_KEY` in `parent_context` for child spans (executor, evaluator), so they export without `lmnr.association.properties.trace_type` and the app-server classifies the trace as `DEFAULT`.

## deepagents instrument

- `Instruments.DEEPAGENTS` is auto-enabled when `deepagents` is installed. When auto-enabled, `LANGCHAIN` and `LANGGRAPH` are auto-removed from the default instrument set (see `_DEEPAGENTS_NOISE_CONFLICTS` in `tracing/instruments.py`) — the LangSmith-style node-level spans they emit add no signal on top of what `LaminarMiddleware` already captures, and clutter the transcript view.
- **Deepagents wins over pydantic_ai when both are installed**: `init_instrumentations` normally strips the raw LLM-provider instrumentors (`OPENAI`, `ANTHROPIC`, …) when `pydantic_ai` is auto-enabled (`_PYDANTIC_AI_PROVIDER_CONFLICTS`), but the deepagents instrumentation relies on those raw instrumentors to emit LLM spans underneath each `TOOL` span — without them the `deep_agent` trace shows only root + tool spans and no LLM children. When `_deepagents_installed()` is true, `init_instrumentations` skips the pydantic_ai conflict-removal so the providers stay enabled. Users with both libraries who want pydantic_ai's de-dup back can either block deepagents (`block_instruments={Instruments.DEEPAGENTS}`) or pass an explicit `instruments` set.
- Root span (`deep_agent`, type DEFAULT) is opened by wrapping the compiled graph's `invoke`/`ainvoke`/`stream`/`astream` at the instance level in `DeepagentsInstrumentor._wrap_graph_methods`, NOT from `AgentMiddleware.before_agent`/`after_agent`. LangGraph runs each middleware hook as its own graph node (a separate task), so OTel context attached in `before_agent` is popped before downstream tool nodes execute and the root span fails to parent subsequent tool spans (symptom: tool spans emit with `parent_span_id=0` on a new trace_id). Wrapping the graph entrypoint keeps the context active for the whole execution.
- `Pregel.invoke` internally calls `self.stream`, so wrapping both without a guard produces two root spans per top-level call. A `contextvars.ContextVar` sentinel (`_root_active` in `deepagents/instrumentor.py`) collapses that to one.
- Subagent cards in the frontend transcript are derived from `lmnr.span.prompt_hash` fingerprinting (see `computeSubagentBoundaries` in `frontend/components/traces/trace-view/store/utils.ts`) — no dedicated subagent span type is needed. A TOOL span named `task` nesting the subagent's LLM/tool spans is enough; the frontend groups them automatically.
- `deepagents` depends on `langchain>=1.0`. `DeepagentsInstrumentorInitializer` returns `None` unless both `deepagents` and `langchain` are installed, so the instrumentor is a silent no-op in environments where only one is present.
- Span-lifetime split: the `invoke`/`ainvoke` and tool-call wrappers use `Laminar.start_as_current_span(...)` as a context manager — the span lives for exactly the `with` block. `stream`/`astream` cannot — the wrapper returns before the caller iterates, so a `with start_as_current_span(...)` around the wrapper body would end the span before any chunk is yielded. Use `Laminar.start_span(...)` to mint the span imperatively *inside* the generator, then re-activate it with `Laminar.use_span(span, end_on_exit=True)`; that handles the 3-way context attach (OTel global context, Laminar isolated context, TracerWrapper span stack), lets `@contextmanager`'s built-in `GeneratorExit` handling leave the span status as UNSET on `break`, and ends the span when the generator closes. Do NOT hand-roll a `_SpanHandle`-style wrapper for this — `use_span(..., end_on_exit=True)` is the public API and is what every other lmnr-python instrumentor uses.
- Stream wrappers (`_wrap_graph_stream` / `_awrap_graph_stream`) must open the span inside the returned generator, not eagerly in the outer function: if the caller never iterates, an eagerly-opened span would never be ended. Additionally, the `_root_active` sentinel must NOT be mutated inside a sync generator body — Python leaks ContextVar changes from sync generators to the caller's context on `yield`, which would make a second concurrent `graph.stream()` call short-circuit and skip instrumentation. The sentinel only needs to be set by `_wrap_graph_invoke` (in its plain-function frame) to collapse `Pregel.invoke → self.stream` into one root span.

## Langfuse bridge

- `Instruments.LANGFUSE` is **auto-enabled by default** when `langfuse >= 3.0` is installed. When auto-enabled, the overlapping raw-provider instrumentors — `OPENAI`, `ANTHROPIC`, `GOOGLE_GENAI`, `GROQ`, `MISTRAL`, `COHERE`, `BEDROCK`, `LANGCHAIN` — are auto-removed (see `_LANGFUSE_PROVIDER_CONFLICTS` in `tracing/instruments.py`) because Langfuse's own `langfuse.openai`, `langfuse.langchain`, and `@observe` wrappers already emit GenAI/tool spans. Deepagents still wins over Langfuse (same reasoning as pydantic_ai): deepagents needs the raw providers to emit LLM children under tool spans.
- The bridge lives in `src/lmnr/opentelemetry_lib/opentelemetry/instrumentation/langfuse/__init__.py` and has two parts:
  1. `LangfuseInstrumentor.instrument(lmnr_tracer_provider, lmnr_span_processor)` attaches Laminar's `SpanProcessor` to every Langfuse-owned `TracerProvider` by iterating `LangfuseResourceManager._instances` and monkey-patching `_initialize_instance` so Langfuse clients created AFTER the bridge installs also get dual-attached. `_handled_providers: set[int]` (keyed by `id()`) prevents double-attach; the Laminar provider itself is skipped.
  2. `LangfuseAttributeTranslator` is a `SpanProcessor` that mutates Langfuse-scoped spans on `on_end` — rewriting `langfuse.observation.model.name` → `gen_ai.request.model`/`gen_ai.response.model`, `langfuse.observation.usage_details` → `gen_ai.usage.{input,output}_tokens`/`llm.usage.total_tokens`, `langfuse.observation.cost_details` → `gen_ai.usage.{input,output,}_cost`, `langfuse.observation.input/output` → `lmnr.span.input/output`, and observation-type mapping (`generation|completion|embedding` → `SPAN_TYPE=LLM`, `tool` → `SPAN_TYPE=TOOL`). Trace-level attrs `session.id` / `user.id` / `langfuse.trace.tags` / `langfuse.trace.metadata.*` map into `lmnr.association.properties.{session_id,user_id,tags,metadata.*}`.
- `LangfuseInstrumentor` does NOT extend `BaseInstrumentor` because `BaseInstrumentor.instrument()` ignores kwargs other than `tracer_provider`/`logger_provider`, and the bridge needs the caller-supplied Laminar `SpanProcessor`. `init_instrumentations` has a special-case branch (`if instrument == Instruments.LANGFUSE: instrumentor.instrument(lmnr_tracer_provider=..., lmnr_span_processor=...)`).
- `LangfuseInstrumentor` idempotency state (`_installed`, `_translator`, `_original_initialize_instance`, `_lmnr_span_processor`) is stored **on the class, not the instance** — both `Laminar.connect_to_langfuse()` and `init_instrumentations` construct a fresh `LangfuseInstrumentor()` each call, so `self.X = v` would create an instance attribute that shadows the class default and every new instance would read the unshadowed `_installed = False`, duplicating the translator processor and layering monkey-patches. Always assign via `type(self).X = v` (or `LangfuseInstrumentor.X = v`) in `instrument`/`uninstrument`/`_patch_resource_manager`/`_unpatch_resource_manager`. Reads via `self._X` are fine because attribute lookup falls through to the class.
- `LangfuseAttributeTranslator.on_end` cannot use `span.set_attribute(...)` to inject the translated attrs: (a) `on_end` is called with an OTel `ReadableSpan`, which has no `set_attribute` method, and (b) even when the object happens to be the recording `Span`, its `end()` has already run so `set_attribute` is a silent no-op. The translator writes directly to `span._attributes[k] = v` — safe because `ReadableSpan._attributes` is the *same dict object* as the recording `Span._attributes`, so the exporter sees the mutation when it serializes the span.
- Public API: `Laminar.connect_to_langfuse()` manually installs the bridge — only needed when `Laminar.initialize(instruments={...})` omits `Instruments.LANGFUSE`. Returns `True` on success, `False` if Laminar isn't initialized or `langfuse` isn't importable. Idempotent (the class-level `_installed` flag collapses repeat calls to no-ops).
- The test suite blocks `Instruments.LANGFUSE` in `tests/conftest.py` for the same reason it blocks `PYDANTIC_AI`: dev deps include `langfuse==3.14.6`, and without the block the `_LANGFUSE_PROVIDER_CONFLICTS` auto-removal would strip OPENAI/ANTHROPIC/etc. from the default set and break every raw-provider test.
- Trace-level attrs in langfuse 3.x live on the **root** span only — `session.id` / `user.id` are NOT inherited onto children by Langfuse itself. The translator applies them only where they already exist (root span). Tests in `test_langfuse.py` exercise each translation branch against a real `Langfuse()` client + `InMemorySpanExporter`.
- Langfuse emits usage/cost as JSON-encoded strings over OTel (`langfuse.observation.usage_details = '{"input":10,...}'`) because OTel attribute values can't be dicts. The translator `_parse_json`s them; it also falls back gracefully if the value is already a dict (shouldn't happen at on_end, but cheap to handle).

## pydantic_ai instrument

- `Instruments.PYDANTIC_AI` is **auto-enabled by default** when `pydantic-ai-slim` (or `pydantic-ai`) is installed. When auto-enabled, the overlapping raw-provider instrumentors — OPENAI, ANTHROPIC, GOOGLE_GENAI, GROQ, MISTRAL, COHERE, BEDROCK — are auto-removed from the default set so the same model call isn't traced twice (pydantic_ai emits its own GenAI spans at the model abstraction layer). The exact set lives in `_PYDANTIC_AI_PROVIDER_CONFLICTS` in `src/lmnr/opentelemetry_lib/tracing/instruments.py`.
- To opt out of the auto-enable, pass `disabled_instruments={Instruments.PYDANTIC_AI}` to `Laminar.initialize`. To keep both pydantic_ai and the raw SDK instrumentors active (accepting duplicate spans), pass an explicit `instruments` set that includes both.
- Installation: `pip install lmnr pydantic-ai-slim>=1.0` — there is no `[pydantic-ai]` extra in `pyproject.toml`.
- The instrumentor does two things:
  1. Calls `Agent.instrument_all(InstrumentationSettings(...))`, which sets a module-level default on pydantic_ai's `Agent` class so subsequent `Agent()` instances pick it up automatically.
  2. Monkey-patches `pydantic_ai.models.instrumented.InstrumentationSettings.__init__` so that *every* construction of `InstrumentationSettings` — including ones triggered by user code calling `Agent.instrument_all(True)` or `Agent(instrument=True)` — defaults to our tracer provider and uses semconv `version=5` when the caller didn't pick one (or picked the legacy `version=1`). User-supplied `tracer_provider=` is respected; a caller-supplied `version >= 2` is also respected unchanged, so callers pinning to an older-but-still-supported semconv contract keep their choice.
- `_instrument` performs no version/installation guards of its own: `BaseInstrumentor.instrument()` already enforces the `instrumentation_dependencies()` constraint (`pydantic-ai-slim >= 1.0.0`), and `PydanticAIInstrumentorInitializer` in `_instrument_initializers.py` only returns this instrumentor when `pydantic-ai-slim` or `pydantic-ai` is installed.
- The test suite runs with `disabled_instruments={Instruments.PYDANTIC_AI}` in `tests/conftest.py` so the raw-provider instrumentor tests still operate on the SDK-level spans they assert on.
- `_instrument` also wraps `pydantic_ai.agent.Agent._run_span_end_attributes` to strip the `pydantic_ai.all_messages` attribute (and the v1-equivalent `all_messages_events`) from the parent agent run span. pydantic_ai dumps the full message history of a run on that span as a JSON blob, but the same content is already on the per-step `chat {model}` child spans via `gen_ai.input.messages` / `gen_ai.output.messages`, so keeping it on the parent doubles ingestion + storage cost. The accompanying `logfire.json_schema` entry is updated in lockstep so backends that read it stay consistent. Tests live in `tests/test_pydantic_ai_settings_patch.py` (`test_run_span_end_attributes_strips_duplicate_messages` / `..._preserves_other_attrs`).

## Temporal instrumentor

The Python Temporal integration (`opentelemetry/instrumentation/temporal/`) is the port of the TS `lmnr-ts` Temporal integration. The shapes are similar, but the wiring is fundamentally different because Python has no ESM module-isolation problem — there is **no `instrumentModules` / `workflowBundle` user wiring**. It auto-registers like every other instrumentor (`Instruments.TEMPORAL`, gated by `is_package_installed("temporalio")`).

- **ONE interceptor covers all three paths.** `LaminarTracingInterceptor` implements both `temporalio.client.Interceptor` AND `temporalio.worker.Interceptor` (`intercept_client` / `intercept_activity` / `workflow_interceptor_class`). The Temporal worker auto-inherits client interceptors (`temporalio.worker._worker` prepends client interceptors to worker ones), so injecting it once at the client level covers client + activity + workflow. This is why TS's three-way split (client patch, worker patch, bundled workflow module) collapses into one object here.
- **Auto-injection patches `Client.__init__`, not `Client.connect`.** `Client.connect` builds config then calls `cls(...)`, so patching `__init__` (`_wrap_client_init` in `temporal/__init__.py`, via `wrap_function_wrapper`) covers BOTH the high-level `connect` path and direct `Client(service_client, ...)` construction. The wrap appends a `LaminarTracingInterceptor` to `kwargs["interceptors"]` only if one isn't already present (idempotent — guards user-supplied interceptor + re-entrant construction). Injection must happen before `_init_from_config` builds `self._impl` (which iterates interceptors in reverse).
- **`workflow_interceptor.py` is sandbox code — it must NEVER import `lmnr`.** The workflow interceptor CLASS (not instance) is returned from `workflow_interceptor_class` and instantiated INSIDE the Temporal deterministic sandbox (`_workflow_instance.py`). `temporalio` is a sandbox passthrough module; `lmnr` is NOT (it pulls in httpx → urllib, which the sandbox blocks with `RestrictedWorkflowAccessError`). So this file imports only `contextvars`, `temporalio.*`, and the pure-constant `.consts`; it creates NO spans, uses NO real uuid/time, and ONLY forwards already-serialized trace headers. `temporalio.workflow` appears solely in string annotations (deferred via `from __future__ import annotations`).
- **Handler header scoping uses a `ContextVar`, not instance state.** Signal/update handlers run as separate asyncio tasks (which COPY the context), so `_handler_headers` (a `ContextVar`) keeps each handler's trace headers isolated from the main workflow path and from concurrent handlers — replay-safe and deterministic. `_active_headers()` returns the handler scope if set, else the workflow-start headers. A signal/update arriving with NO injected trace context must NOT enter the scope (`_has_trace_headers` guard), or `_active_headers` would resolve to an empty map and drop the workflow-start trace from activities scheduled inside the handler. Outbound `_merge` is caller-wins: `{**active_headers, **caller_headers}`.
- **Two headers propagate: `x-lmnr-span-context` (preferred) + `traceparent` (W3C interop)**, both `json/plain` Temporal Payloads via `PayloadConverter.default`. `restore_context_from_headers` prefers the full Laminar header, falls back to traceparent.
- **Debug-context propagation is FREE — there is no `pushLaminarContext` (unlike TS).** On the Python span path, `Laminar.start_span(parent_span_context=restored)` already (a) parses the nested `debug` block and arms the downstream debug runtime, and (b) stamps the `lmnr.span.path` / ids-path parent attributes. So the activity interceptor just hands the restored `LaminarSpanContext` to `parent_span_context=` and both trace nesting and debug propagation happen automatically.
- **`create_activity_span=False` still activates the propagated context — it does NOT make in-activity spans detached roots.** When the wrapper span is disabled, `execute_activity` runs the activity inside `with Laminar.use_span_context(restored):` so manual `observe` calls and auto-instrumented LLM spans created inside the activity still nest under the workflow / client trace (and a propagated `debug` block still arms the runtime). The flag only suppresses the *wrapper* span named after the activity, never the parent-context activation. `Laminar.use_span_context` (`sdk/laminar.py`) is the no-wrapper analogue of `use_span`: it parses the `LaminarSpanContext`, arms debug, pushes a `NonRecordingSpan` parent onto both the OTel context and Laminar's isolated context stack, and calls `processor.set_parent_path_info` so child dotted paths are correct — but creates no span of its own. This was a shared bug with TS (`createActivitySpan: false` was treating in-activity spans as roots); both SDKs now nest. Covered by `test_activity_disabled_still_nests_inner_spans` (inner span shares the propagated trace_id and parents to the remote span id).
- **The workflow-lifecycle span ends on the FIRST terminal handle call.** `_wrap_workflow_handle` shadows the handle's `result` / `cancel` / `terminate` bound methods on the instance (`WorkflowHandle` has `__dict__`) and closes the span once on whichever resolves/raises first. `cancel`/`terminate` also open their own short child spans.
- **OTel span ids are 64-bit but `LaminarSpanContext` stores them padded into a UUID.** A full `uuid.uuid4()` span_id (128-bit) is out of OTel's 64-bit range → INVALID SpanContext that `start_span` silently DROPS (no parent inheritance, fresh trace). Tests that assert parent-trace inheritance must mint a valid id: `uuid.UUID(int=uuid.uuid4().int & ((1 << 64) - 1))` — see the `_span_context()` helper in `tests/test_temporal_instrumentation.py`. (`DebugContext.session_id` / `replay_trace_id` are `str | None`, not UUID — wrap test uuids in `str(...)`.)
- **Tests are pure interceptor unit tests, no Temporal server.** `tests/test_temporal_instrumentation.py` drives the interceptors directly with `SimpleNamespace` fake inputs + fake `next` interceptors, so no server/sandbox is needed (the sandbox-import constraint can't be exercised by unit tests — validate that with a `WorkflowEnvironment.start_local()` scratch script where the workflow + activity live in their OWN lmnr-free module, since the sandbox re-imports the defining module). `temporalio` is pinned in `[dependency-groups].dev`.
